# OPERATOR_HANDBOOK

This handbook is the end-to-end operating procedure for producing one LACN survey cycle release.

## 1) What You Receive

You should receive these artifacts for the new cycle:
- Raw survey CSV export.
- `response_key_YY.csv` or the prior cycle's version to copy and update.
- `progress_YY.csv` or the prior cycle's version to copy and update.
- Optional correction tables (institution/employer/budget overrides).

Core rule: the pipeline runs **local-only**. No Google Sheets dependency is required.

> [!IMPORTANT]
> If the survey questions changed from the prior year, follow `SURVEY_CHANGES_GUIDE.md` **before** proceeding to step 2.

## 2) Stage New-Cycle Files

From repository root `lacn2025/`:

Place raw CSV in `data/` using the naming convention:
```
data/OpsSurveyRawData_YYYY.csv
```

Place metadata in `data/`:
- `response_key_YY.csv` - copy the prior cycle's file, rename it for the new cycle, then update it; see `METADATA_FORMAT_REFERENCE.md` for format
- `progress_YY.csv` - copy the prior cycle's file, rename it for the new cycle, then update it; see `METADATA_FORMAT_REFERENCE.md` for format

Important distinction:
- `response_key_YY.csv` is required cycle input metadata.
- `response_key.csv` without the year suffix is only a pipeline-generated snapshot written after the run.

If needed, add cycle-specific overrides in `data/`:
- `institution_overrides_YY.csv`
- `value_overrides_employer_YY.csv`
- `value_overrides_budget_YY.csv`

See `MAINTENANCE_NOTES.md` §1 for override column specs.

## 3) Register Cycle in Config

Add one row to `config/cycles.csv` for the new cycle. See `CYCLE_ROLLOVER_CHECKLIST.md` §4 for required fields and a copy-paste template row.

Quick verification:

```bash
sed -n '1,5p' config/cycles.csv
```

## 4) Optional: Set Publication Label

If the publish month/year should differ from the current date, set:

```bash
export LACN_PUBLICATION_LABEL="March 2026"
```

The landing page now derives cycle year and response count from the active workspace automatically.

## 5) Run Full Pipeline

Canonical command:

```bash
Rscript code/run_cycle.R --cycle=YY
```

Example:

```bash
Rscript code/run_cycle.R --cycle=26
```

This command runs ingest, cleaning, processing, visual prep, and rendering in sequence and writes `lacn.RData`.

Output behavior on each full run:
- `docs/GeneralReport.html` is overwritten.
- `docs/index.html` is overwritten.
- `docs/custom/*.html` is cleared and fully regenerated by `99_custom_exe.R`.
- Existing historical custom HTML is not archived automatically by the current structure.

## 6) Run Mandatory Quality Gates

Before publish, all gates must pass:
- Schema gate
- Render gate (including custom count parity)
- Cycle output gate
- St. Olaf gate
- Publish artifact gate

Runbook:
- `VALIDATION_GATES.md`

## 7) Run Blocking Validators

Cycle-wide validator:

```bash
Rscript code/validate_cycle_outputs.R --cycle=YY
```

St. Olaf gate:

```bash
LACN_CYCLE_ID=YY Rscript code/validate_stolaf_report.R
```

Expected output:
- `data/cycle_validation_YY.csv`
- `data/stolaf_validation_YY.csv`

Behavior:
- Scripts exit non-zero if any check is `FAIL`.
- Release is blocked until all rows are `PASS`.

## 8) Publish to GitHub Pages

Follow exact git remote and publish steps in:
- `PUBLISH_RUNBOOK.md`

## 9) Release Checklist (Operator Sign-Off)

All items must be true:
- `Rscript code/run_cycle.R --cycle=YY` completed successfully.
- `data/schema_check_YY.csv` exists and has zero failed checks.
- `data/cycle_validation_YY.csv` exists and all rows are `PASS`.
- `docs/GeneralReport.html` exists.
- `docs/custom/*.html` count matches institution count in `lacn.RData`.
- `data/stolaf_validation_YY.csv` exists and all rows are `PASS`.
- `docs/index.html`, `docs/GeneralReport.html`, and `docs/custom/*.html` are present for publish.
- No stale custom HTML files from prior cycle remain in `docs/custom/`.

## 10) If Something Fails

Use:
- `TROUBLESHOOTING.md` for concrete fixes.
- `MAINTENANCE_NOTES.md` for override/schema policy decisions.
- `SURVEY_CHANGES_GUIDE.md` if failures relate to changed survey questions.

## 11) Documentation QA Tests

Run these when updating docs to ensure handoff quality:
1. New-person dry run: a new maintainer follows only `README.md` + this handbook from a clean clone and generates all reports.
2. Rollover simulation: execute checklist steps with mocked next `cycle_id` and confirm no undocumented edits are needed.
3. Gate enforcement test: intentionally break one required field in metadata and confirm docs point directly to diagnosis/recovery.
4. Publish rehearsal: follow `PUBLISH_RUNBOOK.md` and confirm deployed paths match documented URLs.
